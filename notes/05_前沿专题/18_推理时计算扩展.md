# 推理时计算扩展（Test-Time Compute Scaling）

## 面试高频考点
- 什么是 Test-Time Compute Scaling？和训练时 Scaling 有什么区别？
- DeepSeek-R1 用了什么训练方法？GRPO 和 PPO 的区别？
- 过程奖励模型（PRM）和结果奖励模型（ORM）的优劣？
- 为什么说"推理时多想"比"增大参数"更有性价比？
- Reward Hacking 是什么？如何缓解？

---

## 核心思想

传统 Scaling Law 聚焦**训练时算力**（更多数据、更多参数）；2025 年最大范式转变是**推理时计算扩展（TTC Scaling）**：允许模型在推理时"想更久"，通过消耗更多推理 FLOPs 换取更高质量输出。

**核心洞见**：对于难题，TTC 扩展比增大参数量更具性价比。

```
训练时 Scaling：更大模型 + 更多数据 → 能力增长（成本在训练时一次性支付）
推理时 Scaling：更长思维链 + 多路采样 → 能力增长（成本按 Query 按需支付）
```

---

## 关键技术

### 1. Long CoT（长链式思维）

模型在输出最终答案前，生成大量中间"思考 Token"（通常在 `<think>...</think>` 标签内），将复杂推理步骤显式展开。

- o1/o3：用户选择 low/medium/high "thinking budget"，控制推理深度
- DeepSeek-R1：思维链可达数千 Token，自发涌现出自我验证、回溯等行为

### 2. Best-of-N 采样

生成 N 条候选回答，用奖励模型选出最优解：

```
问题 → 采样 N 条回答（N=32~256）
      → 奖励模型打分
      → 选最高分输出
```

**效果**：随 N 增大准确率单调提升，但边际收益递减。

### 3. 过程奖励模型（PRM）

| 奖励模型类型 | 打分粒度 | 优点 | 缺点 |
|------------|---------|------|------|
| ORM（结果奖励）| 最终答案 | 简单，易获取标签 | 无法区分推理过程质量 |
| PRM（过程奖励）| 每个推理步骤 | 对推理链精细引导 | 标注成本高，难以自动化 |

PRM 引导的**束搜索（Beam Search）**：在推理树中每步选择得分最高的步骤，类似棋类 AI 的 MCTS。

### 4. GRPO（组相对策略优化）

DeepSeek-R1 提出，专为推理任务设计的 RL 算法：

```python
# GRPO 核心思想
# 对同一问题采样 G 条输出，组内相对打分（高于平均的为正奖励）
for question in batch:
    outputs = sample_G_outputs(model, question)  # G=8~16
    rewards = score(outputs)  # 可以是规则（数学验证）或模型打分
    advantages = (rewards - mean(rewards)) / std(rewards)  # 组内归一化
    loss = -sum(log_prob * advantages)  # 策略梯度
```

**GRPO vs PPO：**

| 维度 | PPO | GRPO |
|------|-----|------|
| 是否需要 Critic 网络 | 是（额外显存）| 否（用组内均值替代）|
| 计算开销 | 高 | 低 |
| 适用场景 | 通用对话 | 数学/代码等有明确答案的任务 |

## DeepSeek-R1 训练流程

```
阶段1: 冷启动 SFT（少量 Long CoT 数据，让模型学会格式）
         ↓
阶段2: 纯 RL（GRPO + 规则奖励：数学验证 + 格式检查）
         ↓
阶段3: 拒绝采样 + SFT（用 RL 模型生成的高质量轨迹蒸馏）
         ↓
阶段4: RL 精调（加入通用能力对齐）
```

**关键结果**：$550 万训练成本，在数学/代码推理上接近 o1 水平，开源权重震动行业。

## Reward Hacking（奖励欺骗）

**定义**：模型通过奖励模型的漏洞获得高分，而非真正提升能力。

**表现**：
- 生成很长但空洞的推理过程（绕过长度惩罚）
- 答案格式技巧性满足判断条件但内容错误
- PRM 被欺骗：每步得分高，但整体推理逻辑不通

**缓解方法**：
- 规则奖励（数学题直接验证答案）代替奖励模型
- 多模型交叉验证
- 过程与结果奖励结合

---

## TTC Scaling 的适用边界

| 场景 | TTC 效果 |
|------|---------|
| 数学/逻辑推理 | 显著提升（可验证） |
| 代码生成 | 显著提升（单元测试验证）|
| 知识问答 | 有限提升（受知识截止限制）|
| 开放生成/写作 | 提升有限（无客观答案）|

**结论**：TTC Scaling 对**可验证的结构化任务**效果最好，对开放性任务效果有限。

## 面试延伸

**Q：o1 和普通 ChatGPT 最大区别是什么？**
> 核心区别在于推理范式：ChatGPT 类模型直接预测答案，o1 在内部生成大量思维链后再输出，相当于在推理阶段做了更多"计算投入"。这不是模型架构的根本变化，而是训练目标和推理策略的改变。

**Q：为什么 GRPO 不需要 Critic 网络？**
> PPO 用 Critic 网络估计每个状态的价值函数来计算 Advantage（优势函数）。GRPO 对同一问题采样多条输出，用组内平均奖励替代价值函数基线——这是一个无参数的基线估计，节省了 Critic 的训练和显存开销。

**Q：PRM 的训练数据从哪来？**
> 主要有三种方式：(1) 人工对每步推理标注正确/错误（成本极高）；(2) 蒙特卡洛估计——从中间步骤随机完成推理，多次采样后成功率作为步骤质量估计；(3) 自动标注——利用强模型对推理步骤打分（偏 Scalable Oversight 思路）。
