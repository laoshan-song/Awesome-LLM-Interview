## 一、Flash Attention

### 问题：Standard attention 的 IO 瓶颈

标准 Attention 计算的公式虽然简单，但在 GPU 上的内存访问模式非常低效：

```
标准 Attention 的执行流程（每次调用）：

Step 1: 从 HBM 加载 Q, K → 计算 S = QK^T → 将 n×n 的 S 写回 HBM
Step 2: 从 HBM 加载 S → 计算 P = softmax(S) → 将 n×n 的 P 写回 HBM
Step 3: 从 HBM 加载 P, V → 计算 O = PV → 写回 HBM

问题：n×n 的中间矩阵（S 和 P）必须在 HBM 和 SRAM 之间反复搬运
     当 seq_len=8192 时，S 矩阵为 8192×8192 = 67M 个元素
     FP16 下约 134MB，远超 SRAM 容量（A100 SRAM ≈ 192KB/SM × 108 SM ≈ 20MB）
```

```
┌─────────────────────────────────────────────┐
│              GPU 存储层级                     │
│                                              │
│  HBM (High Bandwidth Memory)                 │
│  ├── 容量：40GB/80GB (A100)                  │
│  ├── 带宽：2 TB/s (A100)                     │
│  └── 传统 Attention 在这里读写 n×n 矩阵      │
│       ▲                                      │
│       │ 慢（相对 SRAM 慢 10x+）              │
│       ▼                                      │
│  SRAM (On-Chip Cache)                        │
│  ├── 容量：~20 MB (A100 总计)                │
│  ├── 带宽：19 TB/s (A100)                    │
│  └── Flash Attention 在这里完成全部计算      │
└─────────────────────────────────────────────┘
```

### Flash Attention 的解决方案

**核心思路**：将 Q、K、V 切成小块，每次把一个 Q 块和对应 K、V 块加载到 SRAM 中，在 SRAM 内完成该块的全部注意力计算，只把最终输出写回 HBM。

```
Flash Attention 的执行流程：

1. 将 Q 分成 B_r 个块，K/V 分成 B_c 个块
2. 外层循环（遍历 Q 块）：
   将 Q_i 块加载到 SRAM
   内层循环（遍历 K/V 块）：
     a. 将 K_j, V_j 加载到 SRAM
     b. 计算 S_ij = Q_i × K_j^T（在 SRAM 中）
     c. 用 Online Softmax 更新（不写回 HBM）
     d. 累加 O_i += P_ij × V_j（在 SRAM 中）
   内层循环结束
   将 O_i 写回 HBM（只写一次最终结果）
3. 输出 O，O 在数学上严格等于标准 Attention

关键：n×n 的 S 和 P 从未写回 HBM，永远只在 SRAM 中
```

### Online Softmax（在线 Softmax）

标准 softmax 需要两遍扫描（先求 max，再求 exp sum，最后归一化），这在分块计算中无法直接使用。Flash Attention 通过维护 running max 和 running sum 实现**增量 softmax**：

```
初始化：m = -∞, l = 0, O = 0

对每个块 j：
  1. 计算当前块的 max：m_new = max(m, max(S_ij))
  2. 修正系数：α = exp(m - m_new)
  3. 更新 sum：l_new = α × l + sum(exp(S_ij - m_new))
  4. 更新输出：O = α × (m - m_new) × O + exp(S_ij - m_new) × V_j
  5. 更新 max：m = m_new, l = l_new

最终：O = O / l  （归一化）

数学上等价于标准 softmax，但只需单遍扫描。
```

### Flash Attention 各版本

| 版本 | 年份 | 关键改进 |
|------|------|---------|
| Flash Attention 1 | 2022 | Tiling + Online Softmax，HBM 访问降至 O(N²/M)，训练快 2-4x |
| Flash Attention 2 | 2023 | 改进并行策略（减少非矩阵乘法操作），训练快 2x（vs FA1），达到理论 FLOPs 的 70% |
| Flash Attention 3 | 2024 | H100 专用，利用 WGMMA 指令和 TMA 异步拷贝，达到理论 FLOPs 的 75% |

### 与标准 Attention 的对比

| 维度 | 标准 Attention | Flash Attention |
|------|---------------|-----------------|
| HBM 读写量 | O(N²) | O(N) |
| 中间矩阵存储 | 需要存储 n×n 的 S 和 P | 不存储 |
| 数学等价性 | — | 严格等价（无近似） |
| 反向传播 | 存储中间激活 | Recompute（重计算 S 和 P） |
| 序列长度支持 | 受显存限制（n² 矩阵） | 受速度限制（线性显存） |

## 二、连续批处理（Continuous Batching）

### 问题：Static Batching 的浪费

```
Static Batching（传统方案）：

Batch = [请求A(10 tokens), 请求B(50 tokens), 请求C(20 tokens)]

时间轴：
t=1~10:  A, B, C 都在生成 ✓
t=11~20: A 已完成 → GPU 空等！B, C 在生成
t=21~50: A, C 已完成 → GPU 空等！B 在生成

GPU 利用率随短请求完成而陡降
```

### Continuous Batching（Iteration-Level Scheduling）

```
Continuous Batching：

每步解码后检查：哪些请求完成了？→ 立即移出 → 插入新请求

[请求A ✓完成] → 移出，插入请求D
[请求B 继续] [请求C 继续] [请求D 开始]

GPU 始终保持满载
```

### 影响 Continuous Batching 的因素

1. **请求长度差异**：差异越大，Static Batching 浪费越多，Continuous Batching 收益越大
2. **调度粒度**：越细粒度（逐 token 调度）效率越高，但调度开销也越大
3. **显存约束**：需要保证 KV Cache 有足够空间容纳新插入的请求

## 三、投机采样（Speculative Decoding）

### 核心思路

LLM 推理是 **memory-bound**——生成 1 个 token 和并行验证 k 个 token 消耗的时间几乎相同。因此可以：

1. 用**快但小**的草稿模型（Draft Model）快速生成 k 个候选 token
2. 用**慢但大**的目标模型（Target Model）并行验证这 k 个 token
3. 接受正确的 token，拒绝并重采样错误的 token

### 算法流程

```
输入：前缀 prefix，目标模型 M_q（大），草稿模型 M_p（小）
超参数：候选长度 γ

1. M_p 自回归生成 γ 个候选 token：x̃₁, x̃₂, ..., x̃_γ
2. M_q 并行计算这 γ+1 个位置的前向：
   - 输入 [prefix, x̃₁, ..., x̃_γ]，得到每个位置的输出分布 q(x)
3. 逐 token 验证（Rejection Sampling）：
   对于每个候选 x̃_i：
     a. 从 q(x) 中采样一个 token
     b. 如果采样结果 = x̃_i → 接受
     c. 如果采样结果 ≠ x̃_i → 拒绝，从调整后分布重新采样，停止验证后续候选
4. 将接受的 token 追加到前缀，继续下一轮

输出分布与 M_q 完全一致（数学保证）
```

### 接受率分析

对于每个候选位置 i，接受概率取决于 M_p 和 M_q 的分布差异：

```
如果 M_p 和 M_q 完全一致 → 接受率 = 1 → 加速比 = γ
如果 M_p 和 M_q 完全无关 → 接受率 ≈ 1/词汇量 → 几乎无加速

实际：M_p 通常用同系列的小模型（如 LLaMA-7B 作为 LLaMA-70B 的草稿模型）
接受率 α ≈ 0.7-0.85（取决于任务，代码>通用对话）
预期加速比 ≈ (1-α^(γ+1)) / ((1-α)(γ+平均接受数))
实际加速 2-3x（代码补全更高）
```

### 变体

| 方法 | 草稿模型 | 特点 |
|------|---------|------|
| Original SD | 小号同系列模型 | 需要额外的草稿模型 |
| **Medusa** | 多个额外的解码 Head | 在目标模型上加额外 Head 同时预测多个未来 token |
| **EAGLE** | 特征级预测 | 基于最后一个 token 的 hidden state 预测多个候选 |
| **Lookahead Decoding** | Jacobi 迭代 | 不需要草稿模型，用 Jacobi 迭代生成候选 |

## 四、其他加速技术汇总

| 技术 | 层面 | 思路 | 加速比 |
|------|------|------|--------|
| Flash Attention | 计算 | Tiling + Online Softmax，减少 HBM IO | 2-4x（训练） |
| Continuous Batching | 调度 | 动态插入/移出请求 | 2-10x（服务） |
| Speculative Decoding | 解码 | 草稿模型+目标模型并行验证 | 2-3x |
| Tensor Parallelism | 系统 | 单层矩阵切分多卡，减少单卡计算量 | 近线性（卡数） |
| Quantization (INT8/FP8) | 模型 | 权重压缩，减少显存带宽压力 | 1.5-2x |
| KV Cache Quantization | 存储 | 只压缩 KV Cache，权重不变 | 1.3-1.5x |
| Prefix Caching | 复用 | 多请求共享 system prompt 的 KV Cache | 取决于缓存命中率 |
| Chunked Prefill | 调度 | 长 prompt 分块，与 decode 交错 | 减少 TTFT |
| CUDA Graph | 系统 | 固化计算图，消除 kernel launch 开销 | 10-20% |

## 五、面试延伸

**Q：Flash Attention 和 PagedAttention 是什么关系？**

> Flash Attention 优化的是**单次 Attention 计算的 IO 效率**（怎么算更快），PagedAttention 优化的是**多次 Attention 计算之间的显存管理**（KV Cache 怎么存更高效）。两者解决不同层面的问题，可以（且应该）同时使用。vLLM 同时集成了两者。

**Q：Speculative Decoding 能无限加速吗？**

> 不能。加速上限受限于草稿模型和目标模型的分布相似度。即使分布完全一致，加速比也受限于草稿模型的推理开销。实际中通常 2-3x。此外，草稿模型本身也占显存，增加了系统复杂度。

**Q：Continuous Batching 和 Chunked Prefill 如何配合？**

> Chunked Prefill 将长 prompt 的计算（prefill）拆成多个小 chunk，与正在进行的 decode 步骤交错执行。这避免了单独一个长 prefill 阻塞所有正在 decode 的请求（head-of-line blocking）。vLLM/SGLang 的调度器自动管理这种交错。
