# RAG（检索增强生成）

## 面试高频考点
- RAG 的基本流程是什么？
- 向量检索和关键词检索的区别？
- 如何评估 RAG 系统的效果？

---

## 为什么需要 RAG？

LLM 存在固有局限：知识截止日期、幻觉、无法访问私有数据。
RAG 通过在生成前检索相关文档，将外部知识注入上下文。

---

## RAG 基本流程

```
用户问题
   ↓
1. 检索：将问题编码为向量，在知识库搜索最相似片段
   ↓
2. 增强：将检索到的文档拼接到 Prompt 中
   ↓
3. 生成：LLM 基于文档+问题生成答案
```

---

## 检索方式

| 方式 | 原理 | 优缺点 |
|------|------|--------|
| 稠密检索 | Embedding 向量相似度 | 理解语义，需向量数据库 |
| BM25（稀疏）| 关键词 TF-IDF 改进 | 速度快，无法理解语义 |
| 混合检索 | 稠密 + 稀疏，RRF 融合 | 兼顾两者，生产推荐 |

---

## RAG 的主要挑战与改进

**检索质量差** → Query 改写（LLM 将用户问题改写为更适合检索的形式）

**检索结果含噪声** → Re-ranking（用 Cross-encoder 重排序）

**上下文过长** → 文本分块策略优化、Contextual Compression

---

## RAG vs 长上下文 LLM

| | RAG | 长上下文 LLM |
|-|-----|-------------|
| 知识来源 | 动态检索 | 固定上下文 |
| 推理成本 | 检索成本低 | 随长度增加 |
| 实时性 | 可实时更新 | 受限上下文窗口 |

## 面试延伸

**Q：如何评估 RAG 系统？**
> 检索质量（Recall@K、MRR）+ 生成质量（忠实度 Faithfulness、答案相关性）。RAGAS 是常用自动化评估框架。

**Q：RAG 和 Fine-tuning 如何选择？**
> 知识频繁更新、需要引用来源 → RAG；需要改变输出风格/固定领域知识 → Fine-tuning；两者可结合。
